다른 페이지로 이동/호출
  1. 다른 페이지로 이동
    • Refresh : 응답 페이지를 보여준 뒤 클라이언트가 나중에 다시 요청하도록 한다.
    • Redirect : 응답 페이지 없이 바로 다른 url을 요청하도록 한다.
      • 따라서 redirect의 경우에는 getWriter()를 이용하지 말아야 한다.
  2. 다른 페이지 호출
    • Forward
      • 호출된 페이지로 제어를 넘기기 때문에 해당 페이지가 종료되도 제어가 되돌아오지 않는다. 따라서 forward를 호출하는 측은 getWriter()를 이용하지 말아야 한다. + 때문에 호출자 자신과 동일한 페이지/서블릿을 호출할 수 있다.
      forward는 서버 내에서 요청/응답 객체를 넘겨주므로 별도의 인코딩 없이 텍스트나 다른 객체를 추가하여 넘겨줄 수 있다.
    • Include
      • Forward와 달리 호출된 페이지가 종료되면 제어가 돌아온다. + 때문에 호출자와 동일한 페이지/서블릿을 호출할 수 없다.
      호출된 페이지와 호출한 페이지 사이의 출력 HTML 코드 조각 순서를 맞추기 위해 flush()를 적절히 호출해줘야 한다.

JSP
  1. JSP 파일은 클라이언트 요청으로 필요해지면 HttpJspPage 인터페이스를 구현한 서블릿 클래스로 컴파일되고, 해당 서블릿이 요청을 처리한다
  2. JSP 파일의 본문은 HttpJspPage::_jspService 메서드의 본문이 된다.
  3. HttpJspPage → JspPage → Servlet
    4. JspPage 인터페이스는 jspInit(), jspDestroy()를 정의한다
  4. JSP 태그
    • <%@ %> : 지시자; Directive : page, taglib, include 3종류가 있다
    • <% %> : Scriptlet : 일반 자바 코드를 작성하면 된다. _jspService 메서드 안의 코드로, 매개변수를 포함한 아래의 변수들을 참조할 수 있다.
    • <%! %> : 선언문; Declaration : 서블릿 클래스의 멤버 정의
    • <%= %> : Expression : out::print 또는 out::write에 들어갈 값을 반환하는 문장을 작성하면 된다
데이터 저장소
  1. ServletContext
    2. 애플리케이션 수준 공유 데이터. GenericServlet::getServletContext로 참조할 수 있다. JSP에서는 application 변수.
    initParams : 앱 초기화 매개변수. 모든 서블릿이 ServletContext::getInitParameter로 참조할 수 있다.
    web.xml 예시 <context-param>
    　　<param-name>name</param-name>
    　　<param-value>Context Param</param-value>
    </context-param>
  2. ServletConfig
    3. 서블릿 수준 공유 데이터. GenericServlet::getServletConfig로 참조할 수 있다. JSP에서는 config 변수.
    initParams : 서블릿 초기화 매개변수. 서블릿 내에서 GenericServlet::getInitParameter로 참조할 수 있다.
  3. 당연하지만, 서블릿 static 멤버도 서블릿 수준 공유 가능
  4. HttpSession
    5. 세션 수준 공유 데이터. HttpServletRequest::getSession으로 참조할 수 있다. JSP에서는 session 변수
  5. ServletRequest
    6. (클라이언트 요청 처리) 스레드 수준 공유 데이터. 서블릿마다 자동으로 공유되지는 않으며, 직접 전달해야 한다
  6. 각 클라이언트의 요청이 별개의 스레드에서 실행되므로, ThreadLocal이 유용할 수 있다
  7. JspContext
    8. _jspService()에서 자동 선언하는 pageContext 지역변수(). JSP 실행하는 동안 유효.
javax.servlet 이벤트 리스너
  1. 이벤트 리스너 구현 클래스는 배포 설명자에 서술되거나, @WebListener를 표기하거나, ServletContext::addListener로 등록돼야 한다
    2. @WebListener 표기 가능 리스너 : ServletContextListener, ServletContextAttributeListener, ServletRequestListener, ServletRequestAttributeListener, HttpSessionListener, HttpSessionAttributeListener, HttpSessionIdListener
    ServletContext::addListener 등록 가능 리스너 : ServletContextAttributeListener, ServletRequestListener, ServletRequestAttributeListener, HttpSessionAttributeListener, HttpSessionIdListener, HttpSessionListener. ServletContext 인스턴스가 ServletContainerInitializer::onStartup()으로 설정됐다면 ServletContextListener도 등록 가능
  2. interface ServletContextListener extends EventListener Since Servlet 2.3
    3. default void contextInitialized(ServletContextEvent sce) : 웹 앱 초기화 이벤트
    default void contextDestroyed(ServletContextEvent sce) : ServletContext 종료 이벤트
  3. interface ServletContextAttributeListener extends EventListener Since Servlet 2.3
    4. default void attributeAdded(ServletContextAttributeEvent event) + attributeRemoved, attributeReplaced : ServletContext 속성 추가/제거/변경 이벤트
  4. interface ServletRequestListener extends EventListener Since Servlet 2.4
    5. default void requestInitialized(ServletRequestEvent sre) : 요청이 웹앱 스코프에 진입하면 발생
    default void requestDestroyed(ServletRequestEvent sre) : 요청이 웹앱 스코프를 벗어나면 발생
  5. interface ServletRequestAttributeListener extends EventListener Since Servlet 2.4
    6. default void attributeAdded(ServletRequestAttributeEvent srae) + attributeRemoved, attributeReplaced : ServletRequest 속성 추가/제거/변경 이벤트
  6. interface HttpSessionListener extends EventListener Since Servlet 2.3
    7. default void sessionCreated(HttpSessionEvent se) : 세션 생성 이벤트
    default void sessionDestroyed(HttpSessionEvent se) : 세션 폐기 이벤트
  7. interface HttpSessionAttributeListener extends EventListener Since Servlet 2.3
    8. default void attributeAdded(HttpSessionBindingEvent event) + attributeRemoved, attributeReplaced : 세션 속성 추가/제거/변경 이벤트
  8. interface HttpSessionIdListener extends EventListener Since Servlet 3.1
    9. void sessionIdChanged(HttpSessionEvent event, String oldSessionId) : 세션 ID 변경 이벤트
  9. interface HttpSessionBindingListener extends EventListener
    10. default void valueBound(HttpSessionBindingEvent event) : 이 인터페이스를 구현한 클래스의 인스턴스가 세션의 setAttribute()로 추가되면 호출된다
    default void valueUnbound(HttpSessionBindingEvent event) : 이 인터페이스를 구현한 클래스의 인스턴스가 세션의 removeAttribute()로 제거된 경우, invalidate()로 세션이 만료된 경우, 시간 초과로 세션이 만료된 경우 호출된다.
  10. interface ReadListener extends EventListener Since Servlet 3.1
    11. ServerRequest::getInputStream → ServerInputStream::setReadListener
    void onDataAvailable() : HTTP 요청 데이터 읽기가 가능해지는 시점에 한 번 호출된다
    void onAllDataRead() : 블로킹 없이 모든 요청 데이터를 읽을 수 있을 때 호출된다
    void onError(Throwable t) : 요청 처리에 예외가 발생한 경우 호출된다
  11. interface WriteListener extends EventListener Since Servlet 3.1
    12. ServerResponse::getOutputStream → ServerOutputStream::setWriteListener
    void onWritePossible() : 응답 쓰기가 가능해지는 시점에 한 번 호출된다
    void onError(Throwable t) : non-blocking 쓰기 중 발생한 예외를 처리한다
  12. interface AsyncListener extends EventListener Since Servlet 3.0
    13. ServletRequest::startAsync → AsyncContext::addListener
    void onComplete(AsyncEvent event) : 비동기 작업이 완료되었을 때 호출한다
    void onTimeout(AsyncEvent event) : 비동기 작업이 시간초과된 경우 호출한다
    void onError(AsyncEvent event) : 비동기 작업 중 예외가 발생했을 때 호출한다
    void onStartAsync(AsyncEvent event) : 비동기 작업 시작시 호출한다

    • prototype : getBean() 호출 시마다 생성
    • request : HTTP 요청마다 생성. WebApplicationContext에서만 설정 가능
    • session : HTTP 세션마다 생성. WebApplicationContext에서만 설정 가능
    singleton 빈이 prototype 빈을 참조할 때, 기본적으로는 최초 설정된 prototype 빈 인스턴스만 이용한다. aop:scoped-proxy를 이용하면 singleton 빈이 호출할 때마다 prototype 빈을 다시 생성할 수 있다.

  • 빈 자동 등록
    • @Component : 빈 생성 대상 클래스
    • @Repository : DAO. 퍼시스턴스 클래스
    • @Service : 서비스 클래스
    • @Controller : 컨트롤러 클래스
    • 스코프 설정을 해주고 싶다면 @Scope를 추가로 이용
    • aop:scoped-proxy 설정을 해주고 싶다면 @Scope의 proxyMode 변수를 설정
      • DEFAULT == NO : 프록시 없음, INTERFACES : 인터페이스 프록시 생성. JDK 동적 프록시 이용, TARGET_CLASS : 클래스 프록시 생성. CGLIB 이용
      context:component-scan 태그의 scoped-proxy 속성을 변경하면 DEFAULT 동작을 변경할 수 있다
    • context:include-filter, context:exclude-filter로 포함할/하지 않을 클래스를 구체적으로 명시할 수 있다
      • type 속성 : annotation - 클래스 애너테이션 필터링, assignable - 할당 가능한 타입 필터링, regex - 정규표현식 필터링, aspectj - AspectJ 표현식 필터링
