Getting Started

OpenAPI는 기계적으로 해석할 수 있는 HTTP 기반 API 명세를 작성하는 방법을 정의합니다. OpenAPI를 사용하면 아래와 같은 이점을 얻을 수 있습니다

  • API 요청 및 응답 검증
  • 사람이 읽을 수 있는 문서로 변환
  • 코드 생성 : 다양한 서버/클라이언트 개발 환경에 맞는 API 코드 골격을 자동으로 생성
  • GUI 편집

The OpenAPI Specification Explained

Structure of an OpenAPI Document: JSON, YAML, openapi, info

Document Syntax

OpenAPI 문서는 텍스트 파일로 작성되며, 일반적으로 openapi.json, openapi.yaml를 파일명으로 붙입니다.

Minimal Documnt Structure

OpenAPI 문서는 단일 JSON 객체로 표현되며 포함되는 필드는 OpenAPI 명세(OAS)를 따릅니다. OAS 구조는 매우 길고 복잡하므로 OpenAPI Map과 같은 시각적 도구가 구조를 파악하는데 도움이 될 수 있습니다.

OpenAPI 문서의 루트 객체는 OpenAPI Object이고, 최소 세 개의 필드를 반드시 포함해야 합니다: openapi, info. (paths, components, webhooks 중 하나)

  • openapi (string)

    • OAS 버전을 나타냅니다. e.g. 3.1.0

  • info (Info Object)

    • API에 대한 일반적인 정보를 포함합니다. 아래의 두 개 필드만 필수입니다

    • title (string) : 문서 제목
    • version (string) : 문서 버전
  • paths (Paths Object)

    • 모든 API 엔드포인트 목록을 기술합니다

API Endpoints: paths, response

The Endpoints List

Paths Object는 정의하는 API가 제공하는 모든 작업에 대한 목록을 담는다. Paths Object의 자료 구조는 배열이 아니라 키(path)-값(Path Item Object) 쌍을 갖는 객체인데, 별도의 검증 로직 없이 문맥 검사 단계에서 path의 중복을 방지할 수 있다는 장점이 있다

각 path는 반드시 '/'로 시작해야 한다

↓ yaml

paths: /endpoint1: /endpoint2:

The Path Item Object

Path Item Object는 특정 엔드포인트에 대하여, HTTP 각 메서드(GET, POST, ...)들 중 API가 제공하는 작업(Operation Object)을 정의한다

↓ yaml

paths: /endpoint1: get: put: /endpoint2:

The Operation Object

Operation Object는 작업의 개요와 함께, 요청과 응답 형식을 정의한다

↓ yaml

paths: /endpoint1: get: summary: Do something(short) description: Do something(long) parameters: requestBody: responses: put: /endpoint2:

Content of Message Bodies: content, schema

content 필드는 Response Object와 Request Body Object 모두에 사용되는, RFC6838 Media Types -> Media Type Object 매핑 정보다

↓ yaml

content: application/json: schema: examples: encoding: text/*: ...

Parameters and Payload of an Operation: parameters, requestBody

Parameter Object는 요청 body 외 부분에서 얻을 수 있는 변수를 정의한다

Parameter Location

in 속성을 이용해 파라미터의 위치를 지정한다

  • query : URL 쿼리 문자열
  • header : HTTP 사용자 정의 헤더. 대소문자 구별하지 않음에 유의
  • path : URL 경로

    • ↓ yaml

    paths: /users/{id}: get: parameters: - name: id in: path required: true
  • cookie : 쿠키

Parameter Serialization Control

파라미터의 위치가 다양한 만큼 그 표현 방법도 다양하며, 따라서 직렬화/역직렬화에 필요한 포맷 정보를 style 속성에 설정할 수 있다

  • 원시 타입: 예. 정수 id = 1234
    • style:simpleformlabelmatrix
    1234id=1234.1234;id=1234
  • 배열 타입: 예. ids = [1, 2, 3]
    • style:simpleformlabelmatrix
    explode=false1,2,3ids=1,2,3.1.2.3;ids=1,2,3
    explode=true1,2,3ids=1&ids=2&ids=3.1.2.3;ids=1;ids=2;ids=3
  • 객체 타입: 예. color = {R, G, B}
    • style:simpleformlabelmatrix
    explode=falseR,1,G,2,B,3color=R,1,G,2,B,3.R.1.G.2.B.3;color=R,1,G,2,B,3
    explode=trueR=1,G=2,B=3R=1&G=2&B=3.R=1.G=2.B=3;R=1;G=2;B=3

Reusing Descriptions: components, $ref

API 구성 요소의 재정의를 피하기 위해, 루트 객체의 components 필드에 각 요소들을 정의하고, $ref 속성으로 이를 지정하여 재사용할 수 있다

↓ yaml

# 동일 파일 내 참조 예: $ref: "#/components/schemas/coordinate" # 외부 로컬 파일 내 참조 예: $ref: './another_file.yaml#rowParam' # 원격 파일 내 참조 예: $ref: 'https://gigantic-server.com/schemas/Monster/schema.yaml'

Providing Documentation and Examples: example, examples

YAML에서 긴 설명문을 표기하는 방법

  • Literal mode

    • ↓ yaml

    description: | This is a string in multiple lines. And an extra one.

    ↓ Output

    This is a string in multiple lines. And an extra one.
  • Folded mode

    • ↓ yaml

    description: > This is a string in multiple lines. And an extra one.

    ↓ Output

    This is a string in multiple lines. And an extra one.

Best Practices

  • OpenAPI는 모든 HTTP API를 기술할 수 없다

    • 실제 구현과 문서의 오차를 줄이기 위해, 다소 편법으로 API를 정의할 수도 있을 것이다

  • 대신 OpenAPI는 코드 생성, 문서화, 유효성 검증 등 풍부한 자동화 도구를 제공한다
  • 정보는 다른 곳에 중복하여 기술하면 안 된다

    • 언젠가 한 쪽만 수정된 상황이 발생할 수 있다
