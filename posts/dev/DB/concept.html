DB

  • DB에 저장되는 데이터의 특징

    • Integrated Data, Stored Data, Operational Data, Shared Data

  • DB의 특징

    • Real Time Accessibility, Continuous Evolution, Concurrent Sharing, Content Reference, Independence

    • 논리적 독립 : External schema(View)의 변경 없이 conceptual schema를 변경할 수 있다
    • 물리적 독립 : Conceptual schema 변경 없이 internal schema(물리 장치에 기록되는 형식)을 변경할 수 있다
  • DBMS; DataBase Management System

    • 데이터 정의/조작/제어 기능 지원. 각각은 DDL(Data Definition Language), DML(Data Manipulation Language), DCL(Data Control Language)을 통해 이루어진다

  • Schema : 데이터베이스의 구조와 제약 조건에 관한 명세. 데이터 사전에 저장된다
    • External schema

      • 외부 스키마. 사용자가 필요로 하는 데이터베이스의 논리적 구조

      View : 기본 테이블에서 유도되는, 물리적으로 존재하지는 않는 가상 테이블. SELECT문의 결과

    • Conceptual schema

      • 개념 스키마. 개체 간 관계와 제약 조건을 나타내고 데이터베이스의 접근 권한, 보안 정책 및 무결성 규정에 관한 명세

    • Internal schema

      • 내부 스키마. 데이터베이스의 물리적 구조

키

  1. 각 튜플을 구분하는 기준이 되는 속성들의 집합. 기본적으로 키의 중복은 허용되지 않는다
  2. Super key : 속성들의 집합으로 이루어진 키. 모든 튜플에 대해 유일해야 한다(유일성).
  3. Candidate key : 후보키. 수퍼키 + 최소성 만족(키를 구성하는 속성 중 하나라도 없으면 키로써 작동하지 못하는 상태)
  4. Primary key : 기본키. 후보키 중에서 선택된 키. NULL일 수 없다
  5. Alternate key : 기본키 외의 후보키
  6. Foreign key : 외래키. 다른 릴레이션의 기본키를 참조하는 속성의 집합

무결성

  1. NULL 무결성 : 특정 속성 값이 NULL이 될 수 없는 규정
  2. 고유 무결성 : 각 속성의 값이 유일해야 한다는 규정(둘 이상의 값을 가질 수 없다)
  3. 도메인 무결성 : 속성의 값이 도메인에 속해야 한다는 규정(예. bigint, varchar 등)
  4. 키 무결성 : 하나의 릴레이션에 적어도 하나의 키가 존재해야 한다는 규정
  5. 관계 무결성 : 튜플의 삽입 가능 여부 또는 한 릴레이션과 다른 릴레이션의 튜플들 사이의 관계에 대한 적절성 여부를 지정한 규정. 트리거로도 구현 가능
  6. 참조 무결성 : 외래키 값은 NULL이거나 참조 릴레이션의 기본키 값과 동일해야 한다는 규정
  7. 개체 무결성 : 기본키를 구성하는 어떤 속성도 NULL일 수 없다는 규정

관계대수

  1. 원하는 정보를 어떻게 유도하는가를 기술하는 절차적 언어
  2. SELECT : 특정 조건을 만족하는 튜플의 부분집합을 구한다
  3. PROJECT : 릴레이션의 특정 속성만을 추출
  4. JOIN(⋈) : 공통 속성을 중심으로 2개의 릴레이션을 하나로 . CARTESIAN PRODUCT연산을 수행한 다음 SELECT 연산을 수행한 것과 같다

Anomaly; 이상

  1. 삽입 이상 : 의도치 않게 삽입 불가
  2. 삭제 이상 : 의도치 않게 다른 값들도 삭제
  3. 갱신 이상 : 일부만 갱신되어 정보 불일치

정규화

  • 함수적 종속
    • X → Y : 릴레이션 R의 속성 부분 집합 X, Y에 대하여, 시간에 관계없이 항상 X의 값 각각에 대해 Y의 값 하나만 연관되는 경우
    • 이때 X를 결정자, Y를 종속자라 부른다
    • 완전 함수 종속 : 기본키 전체로 튜플이 결정되는 경우
    • 부분적 함수 종속 : 기본키의 부분만으로 튜플이 결정되는 경우
  • 정규화

    • 테이블을 무손실 분해하여 가능한 한 중복을 제거하여 이상 발생을 줄이는 것. 정규화로 생성된 릴레이션 R1, R2가 NATURAL JOIN으로 손실 없이 R로 복귀되는 경우, R은 R1, R2로 무손실 분해되었다고 한다

    1. 제1정규형 1NF : 모든 속성의 도메인이 원자 값만으로 되어 있다
    2. 제2정규형 2NF : 제1정규형 + 기본키가 아닌 모든 속성이 기본키에만 완전 함수 종속
    3. 제3정규형 3NF : 2정규형 + 기본키가 아닌 모든 속성이 기본키에 대해 이행적 함수 종속을 만족하지 않음
    4. BCNF : 모든 결정자가 후보키인 정규형
    5. 제4정규형 4NF : 다치 종속 A->>B가 존재할 경우, 모든 속성이 A에 대해 함수 종속을 만족
    6. 제5정규형 5NF : 모든 조인 종속이 후보키를 통해서만 성립
    7. 역정규화 : 지나친 정규화는 응답 속도를 저하시키므로, 어느 정도는 테이블을 합치는 것

트랜잭션

  • 하나의 논리적 기능을 수행하기 위한 일련의 연산 집합으로, 작업의 단위. 정상적으로 종료될 경우 COMMIT, 아니면 ROLLBACK
  • 특성
    1. 원자성(Atomicity) : 트랜잭선은 모두 반영되든지 아예 반영되지 않아야 한다.(All or Nothing)
    2. 일관성(Consistency) : 트랜잭션의 실행 결과는 데이터베이스의 제약을 해치지 않을 때만 반영된다
    3. 독립성(Isolation) : 트랜잭션은 다른 트랜잭션의 간섭을 받지 않는다
    4. 영속성(Durability) : 트랜잭션의 결과는 영구히 저장되고 반영된다
  • 상태 : 활동(Active), 실패(Failed), 철회(Aborted), 부분 완료(Partially Committed), 완료

회복

    트랜잭션 수행 중 장애가 발생하여 이전의 정상 상태로 복구하는 작업

  1. 연기 갱신 기법 : 트랜잭션이 완료된 후에야 실질적으로 갱신
  2. 즉각 갱신 기법 : 데이터의 변경을 바로바로 기록하고, 로그를 관리
  3. 그림자 페이지 기법 : 그림자 페이지에 바로바로 기록하고 나중에 실제로 반영
  4. 검사점 기법 : 검사점을 로그에 기록. 장애시 마지막 검사점부터 회복 수행
  5. 미디어 회복 기법 : 데이터베이스를 안전한 저장 장치에 덤프해 놓는다

병행 제어

  1. 동시에 여러 개의 트랜잭션을 병행 수행할 때 실행되는 트랜잭션들이 일관성을 파괴하지 않도록 트랜잭션 간 상호 작용을 제어
  2. 잠금(Lock) : 한 트랜잭션에 사용되는 데이터를 다른 트랜잭션이 접근하지 못하게 함
  3. 2-단계 잠금 규약 : 확장 단계(잠금만 가능), 축소 단계(잠금 해제만 가능)
  4. 타임스탬프 순서 기법 : 트랜잭션 발생 시각 순서대로 실행
