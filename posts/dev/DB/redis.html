Redis와 Valkey

  • 고성능 인-메모리 키-값 데이터 저장소를 이용해 캐시, Pub/Sub 등의 기능을 지원한다
  • 기본적으로 인-메모리로 동작하므로 영속성이 없다
    • 고가용성 구성을 통해 커밋된 데이터 손실을 최소화(~최대 수 초)하거나
    • Amazon MemoryDB와 같이 디스크 쓰기까지 완료한 후에 쓰기 성공 ACK를 응답하여 커밋된 데이터 손실을 0으로 줄일 수 있다
  • Valkey는 Redis 상업화를 계기로, 오픈 소스를 유지하기 위해 Redis 7.2.4 -- BSD 라이선스가 적용된 마지막 릴리스를 포크하여 시작된 프로젝트다

Data type

String

  • 임의 이진 문자열을 이용할 수 있다
  • 단일 문자열의 최대 길이는 512MB -- 문자열이 키로 이용되든, 값으로 이용되든 동일하다

Aggregate data type

  • 같은 작업에 대해 2가지 버전을 제공하는 경우가 있다
    • 블로킹 : 값이 비어 있으면 요소가 추가되거나 시간 초과될 때까지 대기
    • 논-블로킹 : 값이 비어 있으면 처리 종료
  • 키가 존재하지 않더라도 커맨드 성공을 보장한다
    • 요소를 추가하는 경우, 요소가 0개인 값을 자동으로 생성해 사용
    • 그 외의 경우, 요소가 0개인 값에 대한 것과 동일한 응답 반환
  • 요소 제거한 결과 값이 비게 되면 자동으로 키를 정리한다 (스트림 제외)

List

  • 기본적으로 삽입 순으로 유지되는 string 연결 리스트 -- 따라서 리스트 중간에 대한 연산은 비교적 느리다
  • 최대 크기 : 2^32 - 1

Set

  • 순서 없는 string 집합
  • 최대 크기 : 2^32 - 1

Sorted set

  • 각 string 원소는 score라 불리는 실수 가중치(정렬키)를 가진다
  • 같은 score 사이에서는 사전순으로 정렬된다
  • 내부 구현에는 스킵 리스트와 해시 테이블 둘 다 이용한다

Hash

  • field -> value 매핑
  • field, value는 모두 string
  • 최대 크기 : 2^32 - 1
  • 개별 필드 수준으로 TTL을 설정할 수 있다

    TTL 만료 즉시 메모리가 해제되지는 않는다. 주기적인 정리 작업에 의해 정리되기까지 대부분의 명령어는 해당 필드를 없는 걸로 취급한다

    • Since Redis 7.4
      • 만료된 필드에 대한 조회가 발생하는 경우에도 메모리를 해제한다
      • HLEN은 특정 필드에 대한 조회가 아니므로, 아직 메모리가 정리되지 않은 필드도 포함된다
    • Since Valkey 9.0
      • 만료된 필드에 대한 조회가 발생해도 메모리를 해제하지는 않는다
      • HLEN은 아직 메모리가 정리되지 않은 필드도 포함한다
      • HRANDFIELD가 샘플링한 필드가 논리적으로 만료된 경우, 빈 응답을 반환한다

Stream

  • append만 가능한 유사 hash
  • 각 stream entry는 시간과 관련된 ID를 갖고, 1개 이상의 field-value 쌍으로 구성된다
  • 발송 후 즉시 삭제하는 Pub/Sub 서비스와 달리, 엔트리는 명시적인 삭제 요청이 없는 한 영구적으로 스트림에 남는다
  • 수동으로 지정한 ID가 이전보다 작으면 추가에 실패한다
    • ID 전체를 *로 지정하는 경우, 서버에서 자동으로 millisecondsTime-sequenceNumber로 할당

      현재 시각이 이전 entry의 시각보다 작은 경우, 이전 entry의 시각을 사용하고 sequenceNumber만 증가

    • 123-* 처럼 sequenceNumber만 자동 할당할 수도 있다
  • 구현에 radix tree를 이용하여 단일 엔트리 읽기에 O(n) 성능을 보장한다 -- 이때 n은 ID의 길이인데, 보통 짧고 고정이므로 상수 시간 조회와 같다
  • 3가지 방식으로 entry를 획득할 수 있다
    • ID(시간) 범위를 이용해 질의 : XRANGE, XREVRANGE
    • tail -f로 로그를 추적하듯 신규 엔트리 획득 : XREAD
    • Pub/Sub처럼 신규 엔트리 당 1번만 처리 : XREADGROUP
      • 소비자 그룹이 전체 상태를 관리한다 : 마지막으로 전달한 엔트리 ID, 각 소비자별 처리 완료 대기 중인 엔트리 목록
        • XREADGROUP 명령은 쓰기를 수반하므로 프라이머리 인스턴스에서 호출해야 한다
        • 여러 소비자에 대한 로드 밸런싱을 수행하므로, 여러 소비자를 이용하면 메시지 생성 순서와 소비 순서가 불일치할 수 있다
      • 소비자는 동일한 이름을 이용함으로써 연결이 도중에 끊기더라도 작업을 이어갈 수 있다
      • 소비자가 영구적으로 실패하는 경우를 대비해, 소비자 그룹은 보류 중인 엔트리들을 다른 소비자에게 할당할 수 있다
      • 엔트리 자체에 문제가 있을 수도 있다 -- XPENDING 출력의 카운터를 이용해 불량 엔트리(dead letter)를 다른 스트림에 넣고 알림을 보내도록 구현할 수 있다

Geospatial

  • 좌표(경도, 위도)를 저장하고 검색할 수 있다
    • 검색 위치에서 가까운 지점 찾기
    • 검색 범위 안에 존재하는 지점 찾기

Bitmap

string을 비트열로 이용하고, 비트 단위 연산을 지원한다

Bitfield

  • string을 비트열로 이용하되, 정수 단위 연산을 지원한다
  • 정수는 최대 i64(부호 있는 64 비트), u63(부호 없는 63비트)까지 지원된다

Probabilistic

HyperLogLog

  • 집합 크기 추정에 이용되는 확률 자료형
  • 최대 12KB를 사용하여 0.81% 표준 오차 제공
  • 최대 2^64개 멤버를 가진 집합의 크기 추정 가능

Bloom filter

  • Valkey : valkey-bloom 모듈 로드 시 이용 가능
  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Bloom filter 구현을 제공한다
    • 최적 해시 함수 수: ceil(-ln(error_rate) / ln(2))
    • 항목당 필요 비트: -ln(error_rate) / ln(2)^2
    • 1% 오류율: 7개 해시 함수, 항목당 9.585비트
    • 0.1% 오류율: 10개 해시 함수, 항목당 14.378비트
    • 0.01% 오류율: 14개 해시 함수, 항목당 19.170비트

Cuckoo filter

  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Cuckoo filter 구현을 제공한다
  • CF.ADD 명령으로 새 필터를 생성할 수 있지만, 원하는 크기로 설정되지 않을 수 있다 -> CF.RESERVE 이용
  • 레디스 구현은 동일한 키를 여러 번 추가하면 여러 번 저장됨에 유의 -> 대부분의 공간이 비어 있지만 필터가 가득찬 것으로 판단될 수 있음

Count-min sketch

  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Count-Min Sketch 구현을 제공한다
  • 요소가 불균등하게 등장하고, 조회하는 카운트가 높은 경우에만 유용하다

    특정 임계값(= error * total_count) 미만의 결과는 무시해야 한다

  • 오류율은 다음을 의미한다: 임계값 미만의 카운트를 가진 요소를 조회한 결과, 자신의 카운트 대신 충돌하는 임계값 이상의 요소의 최소 카운트를 반환할 확률

t-digest

  • Redis 8부터 코어에 기본 내장된다
  • 백분위수 추정을 빠르게 제공하는 확률적 데이터 구조를 지원한다
  • 다른 데이터 타입들과 달리 키가 존재하지 않으면 TDIGEST.ADD 명령은 무시된다

Top-K

  • Redis 8부터 코어에 기본 내장된다
  • K개 최빈값을 찾는 확률적 데이터 구조를 지원한다
    • width = k*log(k)
    • depth = max(5, log(k))
  • Top-K 목록이 변경되는 경우 알림을 받을 수 있다

JSON

  • Valkey : valkey-json 모듈 로드 시 이용 가능
  • Redis 8부터 코어에 기본 내장된다
  • JSONPath를 이용한 선택 및 업데이트를 지원한다

Vector set

  • Valkey : 호환되지 않음. valkey-search 모듈로 벡터 검색 지원
  • Redis 8부터 코어에 기본 내장된다
    • Sorted set과 유사하지만, score 대신 벡터를 저장한다
    • 양자화 수준
      • Q8 : 8비트. 기본값
      • NOQUANT : 최대 정확도
      • BIN : 1비트
    • REDUCE 옵션을 이용해 차원을 축소해 저장할 수 있다

      이 옵션이 실제로 유용한지는 recall@k 측정 테스트가 필요

    • 가장 중요한 명령은 VSIM: 유사도 검색
      • 검색 기준 벡터는 리터럴 또는 집합 내 요소 이름으로 지정할 수 있다
      • 각 요소에 선택적인 JSON 속성을 설정하고 검색 필터에 이용할 수 있다

Time series

  • Valkey 미지원
  • Redis 8부터 코어에 기본 내장된다
    • 실수 값을 수집 시간과 함께 저장한다
    • 조건을 만족하는 모든 시계열을 한 번에 조회하는 TS.M- 명령들을 제공한다
    • 기간 별 집계를 지원한다 : avg, sum, min, max, count, first, last 등
    • 소스 시계열 데이터를 자동으로 압축하여 대상 시계열에 저장할 수 있다

      예. 10초 해상도 시계열에 대해 장기 보존용 1시간 해상도 시계열 압축

    • TS.CREATE 명령으로 새 빈 시계열을 생성한다

      TS.ADD 명령으로 생성 후 샘플을 추가할 수도 있긴 하다 -- 키가 이미 존재하면, 시계열 생성 관련 인자는 모두 무시

클러스터 목표

다음 목표를 중요도 순으로 구현한다

  • 높은 성능과 선형 확장성
    • 프록시 없이 요청 처리, 비동기 복제 사용, 값에 대한 병합 연산 없음
    • Redis의 경우 최대 1000개 노드, Valkey의 경우 최대 2000개 노드
  • 쓰기 안전성

    과반수 프라이머리 노드들이 처리한 쓰기를 최선의 노력으로 유지

  • 가용성
    • 과반의 프라이머리 노드가 도달 가능하고, 도달 불가능한 모든 프라이머리 노드가 적어도 하나의 도달 가능한 레플리카를 보유한다면 클러스터는 유지된다 -- 레플리카가 프라이머리로 승격된다
    • 레플리카가 없는 프라이머리 노드는, 복수의 레플리카를 갖는 프라이머리 노드로부터 레플리카를 하나 양도받는다

구현 부분집합

  • 클러스터는 비-분산 버전의 모든 단일 키 명령을 구현한다
  • 복잡한 멀티 키 연산은 모든 키가 동일한 슬롯에 해시되는 경우에 한하여 구현된다

    해시 태그를 이용해 특정 키들이 동일한 해시 슬롯에 저장되도록 강제할 수 있다

  • 0번 데이터베이스만 사용 가능하다

    Since Valkey 9.0 : 다중 데이터베이스 이용 가능하다

클러스터 프로토콜에서의 클라이언트-서버 역할

  • 모든 노드는 클러스터 버스라 불리는 TCP 버스로 연결되고, 바이너리 프로토콜을 사용해 통신한다

    모든 노드는 다른 모든 노드와 연결된다

  • 노드는 주어진 키를 담당하는 올바른 노드로 명령을 프록시하는 대신, 클라이언트에 리다이렉트를 지시한다. 클라이언트는 이 정보를 유지해 올바른 노드에 직접 요청을 보낸다

쓰기 안전성

  • Last failover wins: 마지막으로 선출된 프라이머리의 데이터가 결국 다른 모든 레플리카를 대체한다
  • 과반수의 다른 프라이머리와 충분한 시간(NODE_TIMEOUT) 동안 통신하지 못한 프라이머리 노드는 더 이상 쓰기를 수락하지 않는다
  • 쓰기 손실이 발생하는 현실적인 시나리오
    • 쓰기가 프라이머리에 도달하여 클라이언트에 응답했지만, 레플리카에는 전파되지 않은 상태에서 프라이머리가 죽고 -- 시간이 흘러 레플리카가 승격된 경우
    • 파티션이 NODE_TIMEOUT 보다 오래 지속되면 과반수 프라이머리 측에서 페일오버 되므로, 그 동안 소수 측에서 수행된 모든 쓰기가 손실될 수 있다

가용성

  • 클라스터에 파티션이 발생하고, 페일오버 가능하다면 NODE_TIMEOUT + 페일오버(보통 1~2초) 이후 다시 가용 상태가 된다

    장애 발생 ~ 복구 중에는 모든 노드가 쓰기 요청을 거부한다

  • 적절한 레플리카가 프라이머리로 페일오버되어야만 클러스터를 유지 가능하다: 소수 노드 장애만 견딜 수 있다

    예. 5개 프라이머리 노드가 모두 단일 레플리카를 가진 클러스터에서 임의 2개 노드가 분리된 경우, 1/9 확률로 클러스터는 사용 불가능해진다

성능

  • 비동기 복제 사용으로 노드는 명시적인 요청이 없는 한 다른 노드의 쓰기 확인을 기다리지 않는다
  • 클라이언트는 노드와 영구적인 연결을 유지하므로 쿼리는 보통 단일 왕복으로 완료된다
  • 멀티 키 명령은 인접 키로 제한되므로, 리샤딩을 제외하면 노드 간 데이터 이동은 불필요하다
  • 따라서 정상적인 상태라면 N개 프라이머리 노드 클러스터에 단일 인스턴스의 N배 성능을 기대할 수 있다

주요 컴포넌트

키 분배 모델

  • 키 공간은 16384개 슬롯으로 분할되며, 각 프라이머리 노드는 부분집합을 처리한다
  • 키를 해시 슬롯에 매핑하는 기본 알고리즘: CRC16(key) mod 16384

해시 태그

  • 여러 키가 동일한 해시 슬롯에 할당되도록 보장하는 방법
  • 단, 해시 슬롯에 대해 리샤딩 진행 중인 경우 처리 불가능할 수 있다

    이러한 경우 -TRYAGAIN 오류 응답

  • {와 } 사이의 부분 문자열만 해시에 이용된다. 정확히는 아래 3가지를 만족하는 경우에만 그러하다
    1. 키가 { 문자를 포함하고
    2. { 오른쪽에 } 문자가 있고
    3. 첫 번째 {와 첫 번째 } 사이에 하나 이상의 문자가 있는 경우
  • Glob 스타일 패턴 최적화

    예. SCAN 0 MATCH {abc}*는 abc에 해당하는 슬롯만 스캔

ASK 리다이렉션

예. 8번 해시 슬롯을 노드 A에서 노드 B로 이전하는 경우, 마이그레이션 하는 동안엔 해당 슬롯의 키가 A 또는 B에 존재하므로 영구적인 MOVE 리다이렉션 대신 일시적인 ASK 리다이렉션이 필요하다

  • 정상 클라이언트
    1. C -> A : 명령

      8번 해시 슬롯은 A가 처리한다고 기억중

    2. C <- A : ASK B

      해당 키는 B에 있음

    3. C -> B : ASKING 명령

      ASKING으로 A를 거쳤음을 확인한 B가 명령 처리

    4. C -> A : 명령

      8번 해시 슬롯은 A가 처리한다고 기억중

    5. C <- A : MOVE B

      해당 슬롯은 완전히 이전됨

      8번 해시 슬롯은 B가 처리한다고 기억

    6. C -> B : 명령
  • 비정상 클라이언트
    1. C -> B : 명령

      8번 해시 슬롯은 B가 처리한다고 기억중

    2. C <- B : MOVE A

      A를 거치지 않았음을 확인한 B는 해당 슬롯의 기본 처리자가 A라고 안내

      8번 해시 슬롯은 A가 처리한다고 기억

    3. C -> A : 명령
    4. ...

Atomic slot migration

  • Since Valkey 9.0
  • 소스 노드는 자식 프로세스를 포크하여 슬롯에 대한 스냅샷 작성 -> 이후 타깃 노드의 모든 레플리카에 복제
  • 스냅샷 복제가 완료되면 부모 프로세스는 누적된 모든 변경 사항 + 새로운 변경 사항을 전송
  • 전송 중인 변경 사항의 양이 임계값 아래로 내려가면, 소스 노드는 최종 동기화를 위해 쓰기 일시 중지
  • 슬롯 동기화가 완료됐다면 타깃 노드는 슬롯 소유권을 인수하고 클러스터에 브로드캐스트
  • 마이그레이션 완료를 통지받은 소스 노드는 해시 슬롯을 정리하고, 클라이언트에 MOVED 리다이렉션 응답

Client-side caching

  • 클라이언트가 로컬 캐시를 이용해 동일한 항목에 대해 서버 대신 캐시에서 읽기 요청을 처리한다

    서버는 적절하게 무효화 메시지를 보낼 뿐, 캐싱과 무효화를 처리하는 것은 클라이언트 (라이브러리) 책임이다

  • 기본 모드: 서버는 캐싱 활성화된 클라이언트가 읽은 키 집합을 추적하여, 업데이트가 발생하면 무효화 메시지를 보낸다
    • 모든 캐시 활성화한 클라이언트가 읽어간 키를 기억하는 것은 메모리를 너무 많이 점유할 수 있다

      따라서 최대 기억 개수를 초과하면 신규 항목 삽입 시 이전 항목에 대해 무효화 메시지를 발송하고 제거한다

    • DB 번호를 무시한다. 클라이언트가 Valkey 2에서 키 foo를 캐시하고, 다른 클라이언트가 Valkey 3에서 키 foo의 값을 변경하면, 무효화 메시지가 발송된다
    • 효율적인 로컬 캐시 운용을 위해, 클라이언트가 캐시 여부를 언제든 켜고 끌 수 있다 (CLIENT TRACKING, CLIENT CACHING)
  • 브로드캐스팅 모드 : 클라이언트가 접근한 개별 키를 기억하지 않고, 단순히 접두사가 일치하면 무효화 메시지를 뿌린다
  • 클라이언트가 추적을 활성화할 때, 다른 연결의 클라이언트 ID를 지정하여 무효화 메시지를 리다이렉션할 수 있다

    무효화 메시지를 받을 클라이언트는 특수 채널을 구독

    ↓ text

    SUBSCRIBE __redis__:invalidate
  • 캐시 버전 불일치 회피
    • 읽기 응답과 무효화 메시지 수신 사이의 시간 차이가 발생하는 경우

      ↓ 문제 상황

      C1 -> S: GET foo C2 -> S: SET foo bar2 C1 <- S: Invalidate foo C1 <- S: "bar1"

      ↓ 회피 예시

    • 연결이 끊기는 경우
      • 연결이 끊어지면 캐시를 정리하도록 한다
      • 무효화 메시지 수신 채널에서 주기적으로 핑을 보내고, 일정 시간 이상 퐁을 수신하지 못하면 연결과 캐시를 정리
      • 로컬 캐시에도 실제 TTL 또는 적절한 최대 TTL을 설정한다

↓ text

BITFIELD key [ GET encoding offset | [ OVERFLOW <WRAP|SAT|FAIL> ] <SET encoding offset value | INCRBY encoding offset increment> ] [...]
  • 문자열을 비트열로 간주하고 정수 GET, SET, INCRBY 연산들 수행
    • GET 서브 명령 실행 시, 현재 문자열을 벗어난 구간의 비트열은 0으로 간주한다
    • 현재 문자열을 벗어난 위치를 쓰는 경우, 필요 최소한의 길이까지 문자열이 확장된다
  • encoding
    • 부호 있는 64비트(i64)까지 가능
    • 부호 없는 63비트(u63)까지 가능
  • offset
    • 0부터 시작하는 위치
    • 정수만 이용하는 경우 bit단위
    • #를 붙이면 encoding 크기 단위
  • OVERFLOW
    • 이후 등장하는 쓰기 연산에 영향을 미친다
    • WRAP : 기본값, 오버/언더플로우 비트 무시
    • SAT : 오버/언더플로우 시 최대/최소값으로
    • FAIL : 작업 생략, nil 반환
  • 반환값
    • SET : 이전 값
    • GET : 현재 값
    • INCRBY : 신규 값

  • 클라이언트 측 캐싱 활성화/비활성화
  • OPTIN : 기본 모드에서, CLIENT CACHING yes 바로 뒤 읽기 명령의 키만 추적
  • OPTOUT : 기본 모드에서, 읽기 명령의 키를 자동 추적. CLIENT CACHING no 바로 뒤 읽기 명령은 예외
  • BCAST : 브로드캐스팅 모드로 추적. 서버는 개별 키를 기억하는 대신 지정된 접두어를 기준으로 무효화 메시지 발송

    • 접두사는 겹치지 않아야 한다: 한 클라이언트가 "foo"와 "foob" 모두를 추적하는 것은 허용되지 않는다
  • NOLOOP : 이 연결이 수정한 키에 대해, 이 연결에는 알림을 보내지 않는다

SCAN

↓ text

SCAN cursor [MATCH pattern] [COUNT count] [TYPE type]
  • cursor가 가리키는 지점부터 약 count개의 키 중, GLOB style pattern에 일치하는 목록을 반환한다
  • COUNT : 유저가 원하는 페이지 크기에 대한 힌트로 사용된다(기본값 10) -- 반드시 count 개가 반환되는 것이 아님. 또, 연속 순회 시 count는 자유롭게 변경해도 된다
  • cursor : 0을 지정하여 순회 시작. 응답 첫 번째 원소로 0이 아닌 cursor가 반환되고, 이를 이용해 나머지를 순회할 수 있다

    ↓ shell

    > keys * 1) "key2" 2) "key" 3) "hello" > scan 0 count 1 1) "2" 2) 1) "key2" 2) "key" > scan 2 count 1 1) "1" 2) 1) "hello" > scan 1 count 1 1) "0" 2) (empty array)
  • MATCH : 패턴 매칭은 페이지가 구성된 후 클라이언트에 응답하기 직전에 수행되므로, 순회가 종료되지 않더라도 키 목록이 비어있을 수 있다
  • TYPE : 값이 type에 해당하는 키만 검색. TYPE 명령이 반환하는 값과 일치해야 한다
  • 서버는 순회 상태를 저장하지 않으므로, 동시 순회 클라이언트 개수에 제한이 없고, 중간에 언제든 순회를 그만둬도 된다
  • 순회 종료 판단 : 순회로 반환된 커서가 0
  • 일부 복합 타입들은 특정 크기를 초과하지 않는 한 해시 테이블이 아닌 하나의 값으로 저장되므로, count보다 많은 목록이 응답될 수도 있다
