Redis와 Valkey

  • 고성능 인-메모리 키-값 데이터 저장소를 이용해 캐시, Pub/Sub 등의 기능을 지원한다
  • 기본적으로 인-메모리로 동작하므로 영속성이 없다
    • 고가용성 구성을 통해 커밋된 데이터 손실을 최소화(~최대 수 초)하거나
    • Amazon MemoryDB와 같이 디스크 쓰기까지 완료한 후에 쓰기 성공 ACK를 응답하여 커밋된 데이터 손실을 0으로 줄일 수 있다
  • Valkey는 Redis 상업화를 계기로, 오픈 소스를 유지하기 위해 Redis 7.2.4 -- BSD 라이선스가 적용된 마지막 릴리스를 포크하여 시작된 프로젝트다

Data type

String

  • 임의 이진 문자열을 이용할 수 있다
  • 단일 문자열의 최대 길이는 512MB -- 문자열이 키로 이용되든, 값으로 이용되든 동일하다

Aggregate data type

  • 같은 작업에 대해 2가지 버전을 제공하는 경우가 있다
    • 블로킹 : 값이 비어 있으면 요소가 추가되거나 시간 초과될 때까지 대기
    • 논-블로킹 : 값이 비어 있으면 처리 종료
  • 키가 존재하지 않더라도 커맨드 성공을 보장한다
    • 요소를 추가하는 경우, 요소가 0개인 값을 자동으로 생성해 사용
    • 그 외의 경우, 요소가 0개인 값에 대한 것과 동일한 응답 반환
  • 요소 제거한 결과 값이 비게 되면 자동으로 키를 정리한다 (스트림 제외)

List

  • 기본적으로 삽입 순으로 유지되는 string 연결 리스트 -- 따라서 리스트 중간에 대한 연산은 비교적 느리다
  • 최대 크기 : 2^32 - 1

Set

  • 순서 없는 string 집합
  • 최대 크기 : 2^32 - 1

Sorted set

  • 각 string 원소는 score라 불리는 실수 가중치(정렬키)를 가진다
  • 같은 score 사이에서는 사전순으로 정렬된다
  • 내부 구현에는 스킵 리스트와 해시 테이블 둘 다 이용한다

Hash

  • field -> value 매핑
  • field, value는 모두 string
  • 최대 크기 : 2^32 - 1
  • 개별 필드 수준으로 TTL을 설정할 수 있다

    TTL 만료 즉시 메모리가 해제되지는 않는다. 주기적인 정리 작업에 의해 정리되기까지 대부분의 명령어는 해당 필드를 없는 걸로 취급한다

    • Since Redis 7.4
      • 만료된 필드에 대한 조회가 발생하는 경우에도 메모리를 해제한다
      • HLEN은 특정 필드에 대한 조회가 아니므로, 아직 메모리가 정리되지 않은 필드도 포함된다
    • Since Valkey 9.0
      • 만료된 필드에 대한 조회가 발생해도 메모리를 해제하지는 않는다
      • HLEN은 아직 메모리가 정리되지 않은 필드도 포함한다
      • HRANDFIELD가 샘플링한 필드가 논리적으로 만료된 경우, 빈 응답을 반환한다

Stream

  • append만 가능한 유사 hash
  • 각 stream entry는 시간과 관련된 ID를 갖고, 1개 이상의 field-value 쌍으로 구성된다
  • 발송 후 즉시 삭제하는 Pub/Sub 서비스와 달리, 엔트리는 명시적인 삭제 요청이 없는 한 영구적으로 스트림에 남는다
  • 수동으로 지정한 ID가 이전보다 작으면 추가에 실패한다
    • ID 전체를 *로 지정하는 경우, 서버에서 자동으로 millisecondsTime-sequenceNumber로 할당

      현재 시각이 이전 entry의 시각보다 작은 경우, 이전 entry의 시각을 사용하고 sequenceNumber만 증가

    • 123-* 처럼 sequenceNumber만 자동 할당할 수도 있다
  • 구현에 radix tree를 이용하여 단일 엔트리 읽기에 O(n) 성능을 보장한다 -- 이때 n은 ID의 길이인데, 보통 짧고 고정이므로 상수 시간 조회와 같다
  • 3가지 방식으로 entry를 획득할 수 있다
    • ID(시간) 범위를 이용해 질의 : XRANGE, XREVRANGE
    • tail -f로 로그를 추적하듯 신규 엔트리 획득 : XREAD
    • Pub/Sub처럼 신규 엔트리 당 1번만 처리 : XREADGROUP
      • 소비자 그룹이 전체 상태를 관리한다 : 마지막으로 전달한 엔트리 ID, 각 소비자별 처리 완료 대기 중인 엔트리 목록
        • XREADGROUP 명령은 쓰기를 수반하므로 프라이머리 인스턴스에서 호출해야 한다
        • 여러 소비자에 대한 로드 밸런싱을 수행하므로, 여러 소비자를 이용하면 메시지 생성 순서와 소비 순서가 불일치할 수 있다
      • 소비자는 동일한 이름을 이용함으로써 연결이 도중에 끊기더라도 작업을 이어갈 수 있다
      • 소비자가 영구적으로 실패하는 경우를 대비해, 소비자 그룹은 보류 중인 엔트리들을 다른 소비자에게 할당할 수 있다
      • 엔트리 자체에 문제가 있을 수도 있다 -- XPENDING 출력의 카운터를 이용해 불량 엔트리(dead letter)를 다른 스트림에 넣고 알림을 보내도록 구현할 수 있다

Geospatial

  • 좌표(경도, 위도)를 저장하고 검색할 수 있다
    • 검색 위치에서 가까운 지점 찾기
    • 검색 범위 안에 존재하는 지점 찾기

Bitmap

string을 비트열로 이용하고, 비트 단위 연산을 지원한다

Bitfield

  • string을 비트열로 이용하되, 정수 단위 연산을 지원한다
  • 정수는 최대 i64(부호 있는 64 비트), u63(부호 없는 63비트)까지 지원된다

Probabilistic

HyperLogLog

  • 집합 크기 추정에 이용되는 확률 자료형
  • 최대 12KB를 사용하여 0.81% 표준 오차 제공
  • 최대 2^64개 멤버를 가진 집합의 크기 추정 가능

Bloom filter

  • Valkey : valkey-bloom 모듈 로드 시 이용 가능
  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Bloom filter 구현을 제공한다
    • 최적 해시 함수 수: ceil(-ln(error_rate) / ln(2))
    • 항목당 필요 비트: -ln(error_rate) / ln(2)^2
    • 1% 오류율: 7개 해시 함수, 항목당 9.585비트
    • 0.1% 오류율: 10개 해시 함수, 항목당 14.378비트
    • 0.01% 오류율: 14개 해시 함수, 항목당 19.170비트

Cuckoo filter

  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Cuckoo filter 구현을 제공한다
  • CF.ADD 명령으로 새 필터를 생성할 수 있지만, 원하는 크기로 설정되지 않을 수 있다 -> CF.RESERVE 이용
  • 레디스 구현은 동일한 키를 여러 번 추가하면 여러 번 저장됨에 유의 -> 대부분의 공간이 비어 있지만 필터가 가득찬 것으로 판단될 수 있음

Count-min sketch

  • Redis 8부터 코어에 기본 내장된다
  • /posts/fundamental/algorithm.html > Count-Min Sketch 구현을 제공한다
  • 요소가 불균등하게 등장하고, 조회하는 카운트가 높은 경우에만 유용하다

    특정 임계값(= error * total_count) 미만의 결과는 무시해야 한다

  • 오류율은 다음을 의미한다: 임계값 미만의 카운트를 가진 요소를 조회한 결과, 자신의 카운트 대신 충돌하는 임계값 이상의 요소의 최소 카운트를 반환할 확률

t-digest

  • Redis 8부터 코어에 기본 내장된다
  • 백분위수 추정을 빠르게 제공하는 확률적 데이터 구조를 지원한다
  • 다른 데이터 타입들과 달리 키가 존재하지 않으면 TDIGEST.ADD 명령은 무시된다

Top-K

  • Redis 8부터 코어에 기본 내장된다
  • K개 최빈값을 찾는 확률적 데이터 구조를 지원한다
    • width = k*log(k)
    • depth = max(5, log(k))
  • Top-K 목록이 변경되는 경우 알림을 받을 수 있다

JSON

  • Valkey : valkey-json 모듈 로드 시 이용 가능
  • Redis 8부터 코어에 기본 내장된다
  • JSONPath를 이용한 선택 및 업데이트를 지원한다

Vector set

  • Valkey : 호환되지 않음. valkey-search 모듈로 벡터 검색 지원
  • Redis 8부터 코어에 기본 내장된다
    • Sorted set과 유사하지만, score 대신 벡터를 저장한다
    • 양자화 수준
      • Q8 : 8비트. 기본값
      • NOQUANT : 최대 정확도
      • BIN : 1비트
    • REDUCE 옵션을 이용해 차원을 축소해 저장할 수 있다

      이 옵션이 실제로 유용한지는 recall@k 측정 테스트가 필요

    • 가장 중요한 명령은 VSIM: 유사도 검색
      • 검색 기준 벡터는 리터럴 또는 집합 내 요소 이름으로 지정할 수 있다
      • 각 요소에 선택적인 JSON 속성을 설정하고 검색 필터에 이용할 수 있다

Time series

  • Valkey 미지원
  • Redis 8부터 코어에 기본 내장된다
    • 실수 값을 수집 시간과 함께 저장한다
    • 조건을 만족하는 모든 시계열을 한 번에 조회하는 TS.M- 명령들을 제공한다
    • 기간 별 집계를 지원한다 : avg, sum, min, max, count, first, last 등
    • 소스 시계열 데이터를 자동으로 압축하여 대상 시계열에 저장할 수 있다

      예. 10초 해상도 시계열에 대해 장기 보존용 1시간 해상도 시계열 압축

    • TS.CREATE 명령으로 새 빈 시계열을 생성한다

      TS.ADD 명령으로 생성 후 샘플을 추가할 수도 있긴 하다 -- 키가 이미 존재하면, 시계열 생성 관련 인자는 모두 무시

Commands

주의사항

  • MGET, MSET과 같은 다중 키 작업 명령은, 모든 키가 동일한 해시 슬롯에 있지 않으면 CROSSSLOT 오류 발생
    • 키를 동일한 해시 슬롯에 배치하기 위해 태그 사용

      예를 들어 키 {profile}:123와 {profile}:124는 동일한 슬롯에 할당된다

    • 각 키를 개별 명령으로 보내되, 파이프라인을 이용

Bitmap

BITCOUNT

↓ text

BITCOUNT key [start end [BYTE|BIT]]
  • 문자열를 비트열로 간주하고 set bit 개수 반환
  • start end
    • 전부 생략 시 전체 문자열을 검사한다
    • 음수를 허용하는 두 인덱스로 표현되는 닫힌 구간 [start, end]으로 탐색 구간 지정 가능
    • Since Valkey 8.0.0 : end 생략 가능
  • BYTE|BIT
    • : 탐색 구간 단위를 BIT로 지정 가능

BITFIELD

↓ text

BITFIELD key [ GET encoding offset | [ OVERFLOW <WRAP|SAT|FAIL> ] <SET encoding offset value | INCRBY encoding offset increment> ] [...]
  • 문자열을 비트열로 간주하고 정수 GET, SET, INCRBY 연산들 수행

    GET 서브 명령 실행 시, 문자열을 벗어난 구간의 비트열은 0으로 간주한다

  • encoding : 부호 있는 64비트, 부호 없는 63비트까지 가능

    • 예. i5(부호 있는 5비트), u4(부호 없는 4비트)

  • offset : 0부터 시작하는 시작 위치. 정수만 이용하는 경우 bit단위, #를 붙이면 encoding 크기 단위
  • OVERFLOW : WRAP(default, 오버/언더플로우 비트 무시), SAT(오버/언더플로우 시 최대/최소값으로), FAIL(실패, nil 반환)

    • OVERFLOW 이후 등장하는 INCRBY 서브 명령에만 영향을 미침에 유의

  • 반환 : SET(이전 값), GET(현재 값), INCRBY(신규 값)

↓ shell

127.0.0.1:27017> bitfield time SET u6 #0 11 SET u6 #1 33 SET u6 #2 55 GET u6 #0 GET u6 #1 GET u6 #2 INCRBY u6 #0 1 INCRBY u6 #1 1 INCRBY u6 #2 1 1) (integer) 0 2) (integer) 0 3) (integer) 0 4) (integer) 11 5) (integer) 33 6) (integer) 55 7) (integer) 12 8) (integer) 34 9) (integer) 56

BITOP operation destkey key [key ...]

BITPOS

↓ text

BITPOS key bit [start [end [BYTE|BIT]]]
  • bit와 일치하는 인덱스를 닫힌 구간 [start, end] 사이(음수 인덱스 가능)에서 구한다
  • BYTE, BIT : 탐색 구간에 사용되는 start, end의 단위 지정
  • Set bit을 못찾은 경우 -1 반환
  • 구간 지정 없이 Clear bit을 못찾은 경우 비트열 길이 반환

    • 문자열을 초과하는 부분을 0 비트열로 간주하기 때문

  • 구간 지정했지만 Clear bit을 못찾은 경우 -1 반환

↓ shell

127.0.0.1:6379> set key "\x00\xff" OK 127.0.0.1:6379> bitpos key 0 (integer) 0 127.0.0.1:6379> bitpos key 0 1 (integer) 16 127.0.0.1:6379> bitpos key 0 1 1 (integer) -1 127.0.0.1:6379> bitpos key 1 (integer) 8 127.0.0.1:6379> bitpos key 1 1 (integer) 8

SCAN

↓ text

SCAN cursor [MATCH pattern] [COUNT count] [TYPE type]
  • cursor가 가리키는 지점부터 약 count개의 키 중, GLOB style pattern에 일치하는 목록을 반환한다
  • COUNT : 유저가 원하는 페이지 크기에 대한 힌트로 사용된다(기본값 10) -- 반드시 count 개가 반환되는 것이 아님. 또, 연속 순회 시 count는 자유롭게 변경해도 된다
  • cursor : 0을 지정하여 순회 시작. 응답 첫 번째 원소로 0이 아닌 cursor가 반환되고, 이를 이용해 나머지를 순회할 수 있다

    • ↓ shell

    127.0.0.1:6379> keys * 1) "key2" 2) "key" 3) "hello" 127.0.0.1:6379> scan 0 count 1 1) "2" 2) 1) "key2" 2) "key" 127.0.0.1:6379> scan 2 count 1 1) "1" 2) 1) "hello" 127.0.0.1:6379> scan 1 count 1 1) "0" 2) (empty array)
  • MATCH : 패턴 매칭은 페이지가 구성된 후 클라이언트에 응답하기 직전에 수행되므로, 순회가 종료되지 않더라도 키 목록이 비어있을 수 있다
  • TYPE : 값이 type에 해당하는 키만 검색.TYPE
  • 서버는 순회와 관련하여 상태를 저장하지 않으므로, 동시 순회 클라이언트 개수에 제한이 없고, 중간에 언제든 순회를 그만둬도 된다
  • 순회의 종료를 판단하는 기준 : 지정 cursor로부터 남은 키의 개수가 count보다 작은가?
  • 일부 복합 타입들은 특정 크기를 초과하지 않는 한 해시 테이블이 아닌 하나의 값으로 저장되므로, count보다 많은 목록이 응답될 수도 있다

↓ text

  • 해시 필드에 대하여, 아래와 같이 -> 표기를 이용해 BY, GET 옵션을 사용할 수 있다

↓ text

  • key가 없으면 nil 반환
  • key의 값이 문자열이면 해당 값 반환
  • 그 외는 에러

GETDEL

↓ text

GETDEL key
  • GET 실행 후 DEL(key가 존재하고, 값이 string일 경우) 실행
  • GET 실행 결과 반환

GETEX

↓ text

GETEX key [EX seconds|PX milliseconds|EXAT unix-time-seconds|PXAT unix-time-milliseconds|PERSIST]
  • GET 실행 후 ttl 변경. GET 실행 결과 반환
  • EX, PX : ttl(상대 시간)
  • EXAT, PXAT : ttl(절대 시각)
  • PERSIST : ttl 제거

GETRANGE

↓ text

GETRANGE key start end
  • key에 대하여, 문자열의 부분 문자열 [start, end] 반환
  • 음수 인덱스 가능. 문자열 길이를 초과하는 구간은 무시

GETSET; Deprecated

↓ text

GETSET key value
  • SET 실행 후, 이전 값 반환
  • : SET 명령의 GET 옵션 사용

INCR

↓ text

INCR key
  • 주어진 키의 정수 1 증가 후 반환

    • 키가 없으면 0으로 간주

  • 값이 signed 64bit 범위 밖이면 에러

INCRBY

↓ text

INCRBY key increment
  • 주어진 키의 정수 increment 증가 후 반환

    • 키가 없으면 0으로 간주

  • 값이 signed 64bit 범위 밖이면 에러

INCRBYFLOAT

↓ text

INCRBYFLOAT key increment
  • 주어진 키의 실수 increment(음수 가능) 증가 후 반환

    • 키가 없으면 0으로 간주

  • 값이 double 범위 밖이면 에러

LCS

↓ text

LCS key1 key2 [LEN] [IDX] [MINMATCHLEN len] [WITHMATCHLEN]
  • 두 문자열의 LCS(Longest common subsequence)를 구한다

  • LEN : 길이만을 원하는 경우
  • IDX : LCS 각 부분 시퀀스의 [시작, 종료] 위치를 역순으로 반환

  • IDX WITHMATCHLEN : LCS 각 부분 시퀀스의 길이도 반환

MSETNX

MSETNX key value [key value ...]

모든 키가 신규인 경우 mset 실행하고 1 반환. 그 외엔 0 반환

PSETEX

↓ text

PSETEX key milliseconds value

밀리초 단위SETEX

SET

↓ text

SET key value [EX seconds|PX milliseconds|EXAT unix-time-seconds|PXAT unix-time-milliseconds|KEEPTTL] [NX|XX] [GET]
  • 원자적으로 key에 대하여 value를 할당한다. 이전에 설정한 값, 유효 기간을 덮어쓴다
  • EX, PX : ttl(상대 시각)
  • EXAT, PXAT : ttl(절대 시각)
  • NX : 신규 키일 때만 설정
  • XX : 중복 키일 때만 설정
  • KEEPTTL : 이전 ttl 유지
  • GET : 이전 값 반환(없으면 nil)

SETEX

↓ text

SETEX key seconds value

"SET key value EX seconds"와 동일

SETNX

↓ text

SETNX key value
  • key가 존재하지 않을 경우에만 value 설정
  • 설정했으면 1, 아니면 0 반환

SETRANGE

↓ text

SETRANGE key offset value
  • key에 대하여, offset(0부터 시작) 위치부터 value의 길이만큼 value로 덮어쓴다
  • offset이 기존 문자열을 초과하는 경우 빈 자리는 \x00으로 채워진다
  • key가 존재하지 않는 경우, 값을 빈 문자열로 간주하고 실행한다
  • 설정 후 문자열 길이를 반환한다
  • GETRANGE와 함께 O(1) 접근 가능한 공유 배열로 이용할 수 있다

↓ shell

127.0.0.1:27017> set msg hello OK 127.0.0.1:27017> setrange msg 8 hello (integer) 13 127.0.0.1:27017> get msg "hello\x00\x00\x00hello"
